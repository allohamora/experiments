# TanStack Start

- Nitro is the part that was really missing in the Vite community. It enables server-side development with WebSockets on Vite, which is unbelievable.
- Nitro also lets me use any HTTP fetch-based server, which is a really cool idea.
- Nitro v3 is in beta now, and it still lacks documentation for important things. For example, custom presets seem to be supported according to [this issue comment](https://github.com/nitrojs/nitro/issues/2973#issuecomment-3681174855), but there is not enough information about them in the docs.
- Nitro does not have a first-class typed API client for now. It has typed `$fetch` work in progress, but the broader typed routes/client story is still tracked in [nitrojs/nitro#2758](https://github.com/nitrojs/nitro/issues/2758).
- Nitro does not have a clear startup option for things like migrations. There is an issue about [async plugins](https://github.com/nitrojs/nitro/issues/915), and another one about using the [Tasks API](https://github.com/nitrojs/nitro/issues/1974) instead of bootstrap logic in plugins, but for now this part still feels unfinished.
- Graceful shutdown does not work properly right now. `srvx` starts the server, but it does not trigger Nitro close hooks. There is an issue about this here: [nitrojs/nitro#4015](https://github.com/nitrojs/nitro/issues/4015).
- Nitro also has modules that feel unnecessary to me, like its [built-in database layer](https://nitro.build/docs/database). I do not really understand why an HTTP framework needs a unified database integration, or out-of-the-box SQLite support, because every project has different database requirements.
- TanStack Start currently has broken prerendering. I opened [an issue about it](https://github.com/TanStack/router/issues/7473), and another person opened a similar one, but they have been ignored for about a week. This is probably because TanStack Start is also still in beta, and the team is focused on the `1.0.0` release.
- `import.meta.dirname` points somewhere else during the prerender step, so it cannot be used for paths that need to resolve from the project root, like Drizzle migrations [TanStack/router#7518](https://github.com/TanStack/router/discussions/7518).
- Server-side and client-side envs work as expected. Client variables are cooked into the build files through `import.meta.env`, while server-side variables are not and are read at runtime from `process.env`. We do not need `dotenv/config`, because variables are automatically injected from `.env`.
- TanStack Start has a really good API, and for this kind of app it feels ahead of Next.js. I can use isomorphic functions here without creating HTTP endpoints manually, and it has a lot of options for loading data, but not all of them are working correctly yet. It also does not have all Next.js features, like the `<Image />` component, so it is not production-ready yet.
